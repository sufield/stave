# Create Your Own Control

This guide is for security engineers who want to add custom checks without editing Go code.

## Mental model

- Observation: normalized snapshot data (`obs.v0.1`) generated by extractors.
- Control: a safety rule in YAML (`ctrl.v1`).
- Violation Event: emitted when an observation violates a control.

You do not hand-write observations. Generate them first, then write controls against known fields.

## Boundary-first rule writing

Start with a boundary, not a long list of conditions.

Examples of boundaries:

- Public access boundary: "must never be internet-readable"
- Encryption boundary: "must always be at rest encrypted"
- Cross-account boundary: "must not be externally accessible"

Then express one expectation that clearly flips between safe/unsafe.

## Copy an existing control and edit 3 parts

Best workflow:

1. Copy a nearby control from `controls/s3/**`.
2. Edit exactly these first:
   - `id`
   - boundary fields in `unsafe_predicate`
   - expectation (`op` + `value`)
3. Keep everything else stable until validation passes.

## Safe naming scheme

ID pattern:

`CTL.<VENDOR>.<CATEGORY>.<SEQ>`

Examples:

- `CTL.S3.PUBLIC.010`
- `CTL.S3.ENCRYPT.007`

Filename pattern:

`<ID>.yaml` (for example `CTL.S3.PUBLIC.010.yaml`)

Keep sequence numbers zero-padded (`001`, `002`, ...).

## Worked example (simplified from existing S3 control)

This example is based on existing public exposure checks in `controls/s3/public/`:

```yaml
dsl_version: ctrl.v1
id: CTL.S3.PUBLIC.010
name: PHI buckets must not be publicly readable
description: Buckets tagged as PHI must never allow public read access.
domain: exposure
scope_tags:
  - aws
  - s3
type: unsafe_state
unsafe_predicate:
  all:
    - field: properties.storage.tags.data-classification
      op: eq
      value: "phi"
    - field: properties.storage.visibility.public_read
      op: eq
      value: true
remediation:
  description: Public read is enabled on a PHI bucket.
  action: Disable public read and enforce Public Access Block.
```

## Minimal workflow

1. Extract

```bash
stave ingest --profile aws-s3 --input ./aws-snapshot --out ./observations.json --now 2026-01-11T00:00:00Z
```

2. Validate

```bash
stave validate --controls ./controls/s3/custom --observations ./obs
```

3. Evaluate

```bash
stave apply --controls ./controls/s3/custom --observations ./obs --now 2026-01-11T00:00:00Z
```

4. Fix

Apply infrastructure changes based on findings.

5. Re-run

```bash
stave apply --controls ./controls/s3/custom --observations ./obs --now 2026-01-11T00:00:00Z
```

## When you will need engineering help

Ask for engineering help when:

- The observation does not contain the field you need.
- You need reasoning not expressible with current operators.
- You need cross-asset joins beyond current snapshot model.
- You need a new extractor/source type.
