# Docs-As-Code

Stave treats documentation as an engineering artifact, not a post-release task.

## Principles

1. Documentation lives in the same repository as code and is reviewed in pull requests.
2. User-visible behavior changes require docs changes in the same PR.
3. CLI command reference docs are generated by sibling `../publisher` tooling.
4. Link validation runs in Stave CI; generation runs in publisher workflows.
5. Generated docs artifacts are published by workflow automation.

## Operational Workflow

### Generate docs

```bash
make docs
make docs-recordings
```

`docs` delegates CLI markdown generation to sibling `../publisher` tooling.
`docs-recordings` delegates recording generation to the sibling `../publisher` workspace.

### Validate docs

Stave CI validates markdown link checks (Lychee). Publisher workflows own deterministic CLI docs generation.

### Publish docs artifacts

The docs workflow publishes docs artifacts (`.docs-content`) for downstream site pipelines.

## Source Of Truth

- CLI behavior and flags: `cmd/stave/cmd/*.go`
- CLI reference docs: generated from those command definitions
- schema contracts: `schemas/*.json`
- user/developer narrative docs: `docs/*.md`, `README.md`, `SECURITY.md`, `CONTRIBUTING.md`
