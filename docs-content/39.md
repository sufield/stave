## S3 Signed Upload Must Restrict Content Types

**Control:** CTL.S3.WRITE.CONTENT.001 (high)

**Misconfiguration:** The signed upload policy does not restrict `Content-Type`, so users can upload arbitrary file types including SVGs with embedded JavaScript or HTML files. When served from the bucket's domain, these cause stored XSS. The restriction belongs in the presigned POST policy conditions, not in bucket policy or CORS.

**Source data**

This control evaluates upload policies extracted from your application code. The observation captures whether content types are restricted in each signed upload endpoint. Save as `upload_audit.json`:

```bash
cat > upload_audit.json <<'EOF'
{
  "endpoint": "/api/upload",
  "policy_conditions": [
    ["eq", "$key", "uploads/user-123/photo.jpg"],
    ["starts-with", "$Content-Type", ""]
  ],
  "notes": "Content-Type starts-with empty string allows any type"
}
EOF
```

**Extract to a Stave observation (`obs.v0.1`)**

The control checks `type eq "s3_upload_policy"`, `properties.s3_upload.operation eq "write"`, and `properties.s3_upload.content_type_restricted eq false`.

```bash
mkdir -p observations

jq -n \
  --argjson audit "$(cat upload_audit.json)" \
'{
  schema_version: "obs.v0.1",
  generated_by: { source_type: "upload-policy-audit", tool: "manual", tool_version: "1.0" },
  captured_at: "2026-03-20T00:00:00Z",
  assets: [{
    id: "upload-endpoint-api-upload",
    type: "s3_upload_policy",
    vendor: "aws",
    properties: {
      s3_upload: {
        operation: "write",
        content_type_restricted: (
          [$audit.policy_conditions[] |
            select(
              (.[1] == "$Content-Type" or .[1] == "Content-Type")
              and (.[0] == "eq" or (.[0] == "starts-with" and .[2] != ""))
            )
          ] | length > 0
        )
      }
    }
  }]
}' > observations/2026-03-20T000000Z.json

jq '.captured_at = "2026-03-21T00:00:00Z"' observations/2026-03-20T000000Z.json > observations/2026-03-21T000000Z.json
```

**Detect with Stave**

```bash
stave init --profile aws-s3

stave apply \
  --observations observations \
  --max-unsafe 12h \
  --now 2026-03-21T12:00:00Z \
  --allow-unknown-input
```

**Fixed source data**

**AWS doc basis for the fix**

Add an exact `Content-Type` condition to the POST policy:

```json
{
  "conditions": [
    {"bucket": "my-bucket"},
    ["eq", "$key", "uploads/user-123/photo.jpg"],
    ["eq", "$Content-Type", "image/jpeg"]
  ]
}
```

Avoid `["starts-with", "$Content-Type", ""]` which allows any content type. If multiple types are needed, generate separate signed policies per type or use `starts-with` with a specific prefix like `image/`.
